## Правила участия в проекте

Спасибо за интерес к проекту **DevOps Lab**! Ниже приведены базовые правила участия.

### 1. Общие принципы

- **Уважение**: относитесь с уважением к другим участникам и их идеям.
- **Прозрачность**: обсуждайте значимые изменения через Issues и Pull Request.
- **Обучение**: цель проекта — учиться, поэтому приветствуются вопросы и предложения.

### 2. Как предложить изменения

1. **Сделайте форк** репозитория или создайте отдельную ветку от `main` (в этом проекте используется ветка `develop` для активной разработки).
2. **Создайте новую ветку** с понятным названием:
   - пример: `feature/add-ci-pipeline`, `fix/dockerfile-typo`.
3. **Выполните изменения** локально.
4. **Напишите понятные коммиты**:
   - используйте осмысленные сообщения коммитов на английском или русском;
   - пример: `Add basic GitHub Actions workflow` или `Настроить Docker Compose для БД`.
5. **Откройте Pull Request** в ветку `develop` или `main` (в зависимости от договорённостей) и опишите:
   - цель изменений;
   - что именно было сделано;
   - как проверить результат.

### 3. Стиль и качество кода

- **Форматирование**: следуйте общепринятым стилям и форматированию для используемого языка/инструмента.
- **Документация**:
  - обновляйте `README.md`, если меняется поведение или структура проекта;
  - добавляйте комментарии только там, где это помогает понять сложную логику.
- **Тесты** (если применимо):
  - добавляйте или обновляйте тесты для нового функционала;
  - следите, чтобы существующие тесты проходили.

### 4. Работа с Issues

- **Перед началом работы** по возможности находите или создавайте Issue с описанием задачи.
- В Issue указывайте:
  - контекст проблемы или запроса;
  - ожидаемый результат;
  - возможное решение (если есть идеи).

### 5. Code Review

- Будьте готовы к обсуждению изменений.
- Относитесь к замечаниям как к возможности улучшить код и повысить качество проекта.
- При внесении правок по итогам ревью добавляйте новые коммиты, не перезаписывая историю (если нет других договорённостей).

### 6. Лицензия и авторство

- Отправляя изменения, вы соглашаетесь с тем, что ваш вклад может быть использован в рамках лицензии проекта.
- Добавляйте себя в соответствующие списки авторов/контрибьюторов, если такие разделы будут добавлены.

---

Если вы не уверены, с чего начать, создайте Issue с пометкой «question» или «help wanted» — вместе найдём оптимальный путь!
